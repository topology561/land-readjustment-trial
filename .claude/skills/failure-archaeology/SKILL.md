# 失敗考古（每條都真踩過、真付過學費）

## 1. 15m 框（W-D.1.3-b 修除）
**症狀**：項三評分值 8~14（未正規化）、退縮 0m/3.5m 下評分值完全相同。**根因**：硬編 30×30 盒取代法定 range 交集；正確量 `_corner_intersection_area` 早已算好卻沒進評分。**通則**：①**對參數變動無反應的值是頭號嫌犯**（幾何量必隨退縮動）；②單情境測試會遮蔽 bug——3.5m 時 G-gate 把錯誤 winner 淘汰、湊巧「對」，**雙情境強制**；③先找「正確的量是否已存在」再造新輪子。

## 2. 端點配對六塊全滅（-a 修除）
**症狀**：新 matcher 全塊「配對失敗」。**根因**：CAD 線畫到未截角尖角、比 BLOCK 邊長 ~3.5m，端點永不重合——但線畫長是**域上必須**（F/負擔要全長）。**通則**：資料長那樣有它的理由，先問「為什麼資料是這個形狀」再改配對法；修法＝共線＋投影重疊，不是去改資料。

## 3. front_idx 假警報（-a 切換前排除）
**症狀**：診斷五塊 front_idx 新≠舊、被當 regression 停機。**根因**：front_idx 是資訊性欄——`true_S_length` 全檔**只寫零讀**、`front_dir` 被 CAD ALLOC 取代、深度/正面長度全來自 CAD 層。**通則**：把變動當 regression 前，**先追消費者**；只寫不讀的欄位不 gate。診斷 gate 要 key 在 live 量（深度/最小面積/winner），不是內部欄位。

## 4. 行號 ≠ 執行序
**症狀**：以「行號 6336 < 12218」推論執行順序。**根因**：6336 在匯入後才被呼叫的函式裡（而且後來證明是死碼）。**通則**：順序用**呼叫圖＋cache 語義**判，永遠不用行號；行號每波漂移，錨點用符號名。

## 5. 解析期陳舊副本
**症狀**：post-pass 就地改了 geom_restore，block meta 早在解析期把 `cutoff_total_area_m2` 複製出去。**通則**：mutate-in-place 必須同步所有 parse-time 副本，或收斂為單一真相源；grep「誰在早期複製了這個值」。

## 6. 繞向擲硬幣（-c.1，本專案最貴的一課）
**症狀**：-c 首跑截角邊=0、項一≡0、R1右錨 1.0→0.6，僅 R5 倖存。**根因鏈**：tc 出生時在正典逐側迴圈裡（知道自己是哪側）卻**沒把側別存進欄位**；下游兩個消費者只好用「距 FRONT 邊 p1/p2」重猜——BLOCK 邊 p1/p2 循**頂點繞向**、與語意左右零耦合＝逐塊擲硬幣。**潛伏因**：range 減錯側三角形＝no-op、項一原是死常數——舊消費者都容忍，**-c 讓側別第一次承重、潛伏病立刻發作**。**通則**：①**出生地入籍**——身分在哪裡確定就在哪裡存欄位，禁止下游從繞向幾何重猜；②改法要警惕「以前沒事」＝可能只是沒人承重；③新消費者上線＝舊潛伏 bug 的引爆器。

## 7. 單測全綠、整合翻船（-c.1 同案）
**症狀**：`_boundary_len_on_line` standalone 13/13 綠，真 app 全滅。**根因**：單測自造 tc（側別隱含正確），量測器被測了、**側別解析對正典的耦合沒被測**。**通則**：整合測必須**從 app 實際資料結構取材**（AST 抽真函式、餵真 schema 含缺欄情形），fixture 自造＝只測了自己的想像。

## 8. 容差過細漏量
**症狀**：R1右 側街量出 0.041m（實際 29.8m）。**根因**：「同一直邊」由方位差 17″ 的兩段組成、直弦離邊界 ~2mm，1e-6 buffer 量不到。**通則**：線上判定容差 ≥1cm；容差是為捨入緩衝、**不是**湊靶旋鈕（勿調容差湊綠）。

## 9. 靜默歸零掩蓋事故（-a W1 與 -c.1 兩度）
**症狀**：截角解析失敗 → `except: return None` → `else 0.0` → 診斷安靜地顯示 0、無人知道壞了。**通則**：見 no-silent-fallback——缺值必警示、指名街廓/端；裸 except 收窄。

## 10. 快取污染
**症狀**：patch 後行為沒變/a' 累積爆炸。**通則**：每次 patch 後清 session_state 與 `__pycache__`；跨 rerun 的可變物件用 deepcopy 隔離；`@st.cache_data` 回舊 schema（如無 `side` 的 tc）→ 消費者不猜、warn＋None，提示清快取。

## 11. UI session 殘留＝隱形變因（W-D.2 J-fix／R2 殘差雙案例）
**症狀**：①J 表 57 槽兩情境全 0（golden 卻綠）；②R2 3.5m 守恆殘差 −1.0 破線、乾淨複驗 −0.42。**根因**：①該輪 session 未依快照重設步驟 E 路況 → `SS_ROAD` 預設（front 8m 新闢/側街非新闢）→ l₁=0、且 G 側街項同源歸零、R1/R4 正街誤計；②session 中途 Tab5 地價變動 → 同 session 兩套參數混算。**通則**：①**UI session＝隱形參數層**——快照才是正典，實跑前逐參數對快照、驗收表附參數指紋；②對參數不敏感的死值（如 J≡0 而 ΣRw 正常）先查餵入端而非引擎；③根治＝headless harness 消費快照（P6），UI 只做域確認。

## 12. 寫死時間參數＝靜默過期（no-silent-fallback 遠親）
**症狀**：SDMX `end_ym="2026-M3"` 寫死——過期後 API 靜默回舊資料、無人察覺。**通則**：時間參數一律動態（當日＋緩衝）或顯式入快照；grep 全檔 `20\d\d-M`／`date(20` 類字面量；「今天能跑」不等於「明年還對」。

## 13. 同名不同量（「最小分配面積」CLAUDE.md vs 域裁§五）
**症狀**：兩份 KL 權威文件用**同一術語「最小分配面積」**指涉不同公式，實值每塊皆異
（UC9898 R4：街角規定範圍 109.2 vs `round(分配深度×min_width,2)`=114.07）。**根因**：
①**街角規定範圍面積**（`_build_corner_range_v2`，退縮+min_width 平移多邊形）＝街角地 PK
資格（W-D.1.3、per-side）；②**最小分配面積標準**（`f3_min_alloc_area_by_label`，D_avg×min_width）
＝地基檢查+四梯達標（W-C §7-0a、W-D.4、per-block、區標準=MIN_i）。CLAUDE.md 舊行誤稱兩者
「同源同值」。**通則**：①load-bearing 術語若被兩處引用、先驗「同名是否同量」——列每塊實值對照，
差>容差即脫鉤；②衝突屬**兩 KL 權威打架**＝判斷層，reviewer 勿自裁、上呈 KL；③裁定後**同步修
CLAUDE.md**（同名不同量並列具名），免後波再撞。碼與域裁本已一致時，outlier 常是自然語言鐵律行。

## 14. 刀口值不猜、進 REPL 驗（float round／banker's rounding）
**症狀**：W-D.4 我聲稱「`round(114.065,2)=114.06` 會 FAIL」→ 選了未 round MinA；KL 自驗推翻＝
`32.59×3.5=114.065000…1`、`round(…,2)=114.07`（float 表示略高於 114.065）。順帶下一刀口：
`round(57.035,2)=57.03`（banker's/float，非 57.04）。**根因**：2dp 邊界值（x.xx5）之 round 結果
**取決於 float 精確表示與 banker's rounding**，肉眼/心算不可靠。**通則**：①**任何 .xx5 刀口值進
REPL 驗、禁口頭斷言**（尤其拿來當「選 A 不選 B」的理由時）；②判定式避開刀口——用**整數倍
比較**（`2×ΣG≥MinA` 取代 `ΣG≥MinA/2`）消滅除法與二次 round；③顯示值需 §法定四捨五入時用
`Decimal(str(x)).quantize(ROUND_HALF_UP)`、**非** float `round()`，並加斷言把刀口釘成閘；
④**觸及鐵律值/load-bearing 常數時即停手上呈**（#13 之延伸：定義衝突上呈、刀口值也上呈，不接力猜）。

## 15. 報表層分類器之輸入域 ≠ 引擎作用域（W-F 缺口D，最貴的一次「靜默 else」）
**症狀**：W-D.4 四梯清單把 8 個**純公設地歸戶**判為「梯3 現金補償」；而 W-F §7-2/7-3/7-4 明文要
把同一批公設地**調配**。同一批群被兩處指派、互相矛盾；且清單之 §53 補償額算成 **0 元**（假值）。
**根因**：分類器 `tier` 的四個分支吃 `ΣG_戶`，而 `ΣG_戶` 只加總**可建築宗**的 G。無可建築宗之歸戶
`ΣG=0` → 前三個 `if/elif` 全不成立 → 被**最後的 `else`（最低梯）靜默吞掉**。`lots` 為空使
`Σa_i×地價` 亦為 0，補償欄印出「0」——**一個看起來合法的數字，其實是「不適用」**。
**通則**：①**分類器上線前先問「輸入域是什麼？域外個體會落到哪個分支？」**——`else` 是預設收容所，
不是「其餘皆屬此類」的證明；②域外個體必須**顯式標記**（本案加 `軌別` 欄：建地軌／公設軌／無地軌，
公設軌 `梯次="—"` 不判），**不得讓 else 吞掉**；③**空集合的聚合值是陷阱**：`sum([])==0`、
`all([])==True`——0 元補償與「全數達標」都是空集合的假訊號，需 `if not lots` 前置守衛；
④跨波引用同一張表當「驗收準源」時，**先驗該表的值域涵蓋所有待驗個體**（本案清單被指定為 F.4
總決算準源，卻在 8/33 群上失效）；⑤取值停機條款：值未算出前，欄位**留空**而非填 0
（`no-silent-fallback` 之表親——靜默 0 比缺值更難抓）。

## 16. 圍欄比對粒度 < 引擎消費粒度（#11 家族新變體；W-G G.2 活抓）
**症狀**：G.2 誠實圍欄比對**均價**（重劃前/後平均地價）全綠，但區段級單價（前區段 a/b、後街廓 R1-R6）漂移——均價層完全看不見。**根因**：均價＝加權和/總量，per-zone 漂移可互抵使均價恆等（區段間搬移不改總量→均價不動）；圍欄只比均價＝**比對粒度粗於真變因粒度**。同 #11「UI session＝隱形變因」家族：圍欄「全綠」給假安心，隱形變因（per-zone 單價）在更細粒度潛伏；亦 #15「空集合/聚合值是陷阱」之表親——聚合值掩蓋成分漂移。**通則**：①誠實圍欄比對粒度必須 **≥ 引擎消費粒度**——引擎 A 逐宗吃 per-zone/per-block 單價，圍欄就得逐區段/逐街廓並列比對，非只比均價；②「圍欄全綠」不證「無漂移」，只證「**所比之粒度**無漂移」，先問「我比的粒度涵蓋引擎真吃的粒度嗎？」；③活抓法＝把引擎實際消費的每個 scalar 拉出逐一並列，非只比其聚合（本案補前區段 a/b＋後街廓六價逐格入圍欄表）。

## 17. 正面案例：no-silent-fallback loud 閘於生產路徑首次兌現（W-G G.1 live 停機）
**症狀（正面）**：G.1 接線後真 app（localhost、非 harness）首跑，`run_step_g` 於 `f3_total_burden_rate_from_finance` 未鋪底時 **loud 停機**（RuntimeError 明指「須先 compute_total_burden_rate()、禁靜默退回 0.40 舊 fallback」，stepg:171-176）。**根因（為何是好事）**：舊碼 `or 0.40` fallback 會靜默用錯率算完、無人察覺（#9/#11 家族）；財務接線波廢 fallback 改 loud。**生產路徑首次踩到此路徑**——harness 因 build_pipeline 主動鋪底而永不觸，故 live 使用面才是第一次承重，閘正確咬合＝把「ctx-builder 合約缺口」在崩潰點指名，而非靜默吐錯數字。**通則**：①no-silent-fallback 的價值在「生產首跑」才完全兌現——是 #6「新消費者＝舊潛伏 bug 引爆器」的正向版（loud 閘讓引爆＝可見停機、非隱形錯值）；②修法＝補鋪底鍵、決定性現算（呼叫同一 `compute_total_burden_rate`、β 快照輸入），**禁**回退 fallback、**禁**要求使用者先點某 Tab（#11 脆弱路徑）；③接線層上線前**掃全鏈**「讀 session 鍵、缺則 raise」之 loud 前置點，逐一確認接線層已鋪或 live 必有（G.1 掃結果：唯一缺口＝此率鍵；餘皆 run_step_g 自寫或 `.get(default)`）——別讓使用者一鍵一鍵踩地雷。

## 18. plotly_events 點選判定綁 curveNumber-by-z＝脆弱（W-G GIS 步驟D跳位；KL localhost 活抓）
**症狀**：步驟 D 重劃前區段標註，點 628-35(2) 卻標到 628-37（ghost 歸屬主體）。**根因鏈**：①ghost hover **marker** trace（`_ghost_hover_unified` 3120／`_ghost_hover` 2353）於「點擊靶 trace」後 add_trace → **z-top＋可點**；plotly_events 咬到它時回傳**該 marker 之點**（curveNumber=ghost trace、ev.x/y=ghost 質心 near 628-37），消費端 else 分支用該 x/y 做 shapely → 解到 628-37。②**更深**：步驟 D 消費端 `valid_parcels` 只濾 `len<3`（**含 ghost**），圖面點擊靶卻排 ghost → `curveNumber−offset` 之 **index 與圖面錯位**（ghost 前置即跳位），即使命中正確靶亦錯。
**當前有效之根治（本波實裝）**：①消費端**只認點擊靶命中**（cn 在 `[click_offset, +N_parcels)`），cn 指向非靶 trace（ghost marker／填色 z-top）時**丟棄**、**禁用其 ev.x/y 做後援**（該 x/y 是 trace 之點、非點擊位置）；②**cn=None 才** x/y shapely 後援；③**渲染集與判定集必須同源**——消費端過濾集須與圖面產生集同準（本案 valid_parcels 同步排 ghost），否則 index 錯位（#3「只寫不讀不 gate」之對偶）。
**⚠️ 血的教訓（reviewer 逐 JS 追證）**：原擬「customdata 白名單（`__clk__<暫編地號>` by 業務鍵、與 z 序全解耦）」為**最理想根治**，然 **`streamlit-plotly-events==0.0.6`（`_RELEASE=True`）前端只轉發 5 鍵（x/y/curveNumber/pointNumber/pointIndex）、不含 customdata** → `ev.get('customdata')` **恆 None**、白名單分支＝**死碼**。**通則**：④**「根治機制」若依賴一個未實證的能力（此處＝事件 payload 帶 customdata），該機制可能是死碼——制度記憶只准記「已驗證生效」者為根治**；跨庫事件轉發**逐鍵驗證 payload**（讀該套件實際轉發碼／實跑印 `ev.keys()`），禁假設「plotly 有 customdata → plotly_events 就轉發」。customdata kwargs 現保留為**前瞻鉤**（inert；待升級套件轉發後方生效，屬 KL 相依決策）。⑤與「預設濾 ghost」(answer-1) 關係：濾 ghost 後預設視圖無此 marker、bug 隱形消失，但 z-序脆弱性仍在（任何後加頂層 trace 重演）→ 上列①②③為根治、非繞過。

## 19. 同類 bug 藏多個獨立消費端·修一處後須全函式掃描（W-G GIS 區段點選第二破口；KL localhost 活抓）
**症狀**：024915b 修完步驟 D 跳位後，KL 沿 R1/R4/R6 ghost 密集帶（628-1(1)/628-35(2) 周邊）逐點點擊，仍有「未對到任何地號／誤配」。**根因**：GIS 有**兩個獨立 plotly_events 消費端**——§2（`_pevents_um` 12441）與步驟 D（`_pevents_d` 12883）。024915b 修的是步驟 D 的 `valid_parcels`（排 ghost）＋兩端 curveNumber 判定路徑；但 §2 的 **cn=None shapely 後援**走 `_resolve_clicked_parcel(cx,cy,temp_parcels)`（5405），其候選收集迴圈**只排 `len<3`、未排 `_is_ghost_sliver`**，而 `temp_parcels` 含 ghost → 容差 buffer 命中 ghost（取面積最大）→ ghost 無自身區段代碼 → 誤配/落空。**此為與 024915b 同一 bug 類（ghost 污染點選候選集）之第二實例、藏在另一消費端**。

**根因**：同一「boolean difference＋buffer 膨脹＋面積過濾」慣用法在**跨波演進**中被**複製到每個新增的幾何重算點**（Step G→F.1 整形→E3 整形），每處都是「照抄 stepg 同式」（註解自承）。CC 靠「grep 主症狀字串於主檔」找病灶，但**整形段用了語意變體**（`unary_union(list(npolys.values()))` vs `_uunion_d(allocated_polys)`·變數名不同）致字串搜尋漏網；補償碼更是**反向操作**（`buffer(+ε).buffer(−ε)`）·與主症狀 `buffer(0.001)` 字面不同、靠讀懂「為何要補償」才追得到。
**通則**：①**修一個機制性病灶前，先問「這個慣用法在專案裡被複製幾次」**——grep 應搜**機制關鍵詞的多個變體**（`buffer(0.001)`／`buffer(0.0011)`／`.difference(`／`area >= 1.0`／`unary_union`）＋**讀每個幾何重算點的註解**（本專案註解常自承「stepg 同式」＝複製鐵證）。②**配套的補償/繞過碼與主病灶是一體**：主病灶產生的假象（1mm 侵蝕縫）往往在下游有「補償碼」（閉運算彌合）——**修主病灶必同步拆補償**，否則補償變過度校正（#5「解析期陳舊副本」之動態對偶：一個 bug 的痕跡散在多處·修源頭須清全部痕跡）。③**「找到兩處」不等於「找齊」**——#19（多消費端）之靜態對偶：#19 是同一污染在多個**呼叫點**重演·本條是同一慣用法在多個**實作點**被複製；二者都要求「修一處後全庫掃描同類」·差別在 #19 掃呼叫圖、本條掃程式碼慣用法變體。④**跨層複驗擋得住此類漏**：CC 出的 plan 若直接施工只會改兩處·§N5 回寫 g_tab 時整形段的舊式漏會「原路返回」——claude.ai 進倉逐檔 grep 才補齊·印證「reviewer 獨立復現＝全價值所在」（CLAUDE.md 鐵律）。
